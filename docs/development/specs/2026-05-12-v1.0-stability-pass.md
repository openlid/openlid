# Open-Lid v1.0 — Stability Pass

**Status:** Draft for implementation
**Date:** 2026-05-12
**Author:** Diyan Bogdanov (with Claude)

## Summary

Lock the user-facing CLI, configuration, and IPC surfaces so v1.x can carry a semver guarantee. Add a `version` field to `config.toml` as a forward-compatibility hook for future v2.x schemas. Remove the v0.3 roadmap entry (skipped — schedule UI, notifications, and hotkey are not planned).

This pass is ~80% policy/documentation and ~20% code: one new field on `Config`, one constant, four tests, and one new doc.

## Goals

1. Publish a durable, public commitment about which Open-Lid surfaces follow semver.
2. Add the minimum versioning hook to `config.toml` so a future v2 schema change can be detected and migrated cleanly.
3. Reconcile the README roadmap with the decision to skip v0.3.

## Non-goals

- **v0.3 features.** Schedule preference UI, optional state-change notifications, and configurable hotkey are explicitly out of scope and removed from the roadmap.
- **JSON Schema artifact files.** The control-protocol surface is small enough to document inline in `docs/COMPATIBILITY.md`.
- **CI snapshot tests for schema drift.** Useful if drift becomes a real issue; not justified at v1.0.
- **Public Rust API stability** for `open-lid-core` or `open-lid-helper-protocol`. These crates have no external consumers; they're internal to the workspace.
- **Config migration code.** Adding `version` at v1.0 is the *hook*; no migration logic is needed because the schema isn't changing in this pass.

## What gets locked

The semver promise covers exactly these surfaces. Anything not listed here may change between any two v1.x releases.

| Surface | Promise (v1.x) | Source of truth |
|---|---|---|
| `open-lid` CLI subcommands, flag names, exit codes | Names and semantics are stable. Additive only — new subcommands and new flags may be introduced; existing ones must not be renamed, removed, or have their semantics changed. | `crates/app/src/cli/mod.rs` |
| `open-lid status --json` output | Field names and value types are stable. New fields may be added; consumers must ignore unknown fields. | `Snapshot` in `crates/core/src/ipc/control.rs` |
| `config.toml` field names | Field names are stable. Removed/renamed fields are breaking. New fields are ignored by old binaries via serde's `#[serde(default)]`. | `Config` in `crates/core/src/config.rs` |
| Control socket JSON (UDS) | `ControlRequest` / `ControlResponse` JSON shapes are stable. Additive only. Clients must ignore unknown fields. | `crates/core/src/ipc/control.rs` |
| Helper XPC protocol method signatures | Method signatures are stable within v1.x. (The helper always ships with its client inside the same `.app` bundle, so version skew is impossible in practice. The promise is still useful for third-party tools that connect to the helper directly.) | `crates/helper-protocol/build.rs` |

## What is NOT locked

- Internal Rust types, module structure, trait shapes, and crate organisation.
- The UDS socket path (`~/Library/Application Support/open-lid/control.sock`).
- Log file paths and log line format.
- The CLI's auto-launch-the-app behavior when the menubar isn't running.
- The control-socket transport (currently line-delimited JSON). The shapes are locked; the framing may evolve.
- Anything in `open-lid-core` not surfaced through `Config` or the IPC types listed above.

## Config schema versioning

Add a `version: u32` field to `Config`:

```toml
# Existing config.toml gains one new line on save:
version = 1
enabled = false
start_at_login = false
# ... rest unchanged
```

Behavior:

- **Load, missing field** → default to `1`. Existing v0.1 configs (no `version` field) round-trip cleanly into v1.0 with no user action required.
- **Load, `version > SCHEMA_VERSION`** → log a warning via `tracing::warn!`, then continue. Serde's forward-compat behavior (ignore unknown fields) handles a v2 config under a v1 binary with degraded behavior, which is the right tradeoff for a personal-use utility: a user who tries a v2 beta and downgrades is not locked out of their config.
- **Save** → always write `version = SCHEMA_VERSION` (currently `1`). Old configs gain the field on first save by the new binary.
- **Constant:** `pub const SCHEMA_VERSION: u32 = 1;` exported from `open_lid_core::config`.

The warn-don't-fail choice is deliberate. The alternative — refuse to load a config whose version is unknown — protects against misinterpretation but is hostile to anyone who tries a beta. Misinterpretation risk is also low at v1.0 because there's nothing to mis-interpret: serde will silently drop unknown fields, which is the conservative behavior.

## Testing

New tests in `crates/core/src/config.rs`:

- **`version_defaults_to_one_when_missing`** — write a `config.toml` with no `version` line, load it, assert `cfg.version == 1`.
- **`save_always_writes_current_version`** — save a `Config { version: 1, .. }`, re-read the raw TOML, assert it contains `version = 1`.
- **`load_with_newer_version_succeeds`** — write a config with `version = 999`, load it, assert it returns `Ok` and that the known fields parsed correctly. We do not assert the warning was emitted (would require adding `tracing-test` as a dev-dep for one assertion); the behavior under test is "doesn't fail".
- **`save_then_load_round_trip_preserves_version`** — extend the existing `save_then_load_round_trip` to assert `cfg.version == 1` round-trips.

## Learning-mode contribution

One genuine design choice is left to the user to implement: the warn-on-newer-version branch in `Config::load`. The function will be prepped with the surrounding code, a clear TODO comment block, and two implementation sketches to choose between:

- **Option A — log-and-continue (recommended in this spec):** call `tracing::warn!` and proceed with the existing parse path. Five lines.
- **Option B — log-and-return-default:** treat an unknown version as "config is from the future, I cannot reason about it, fall back to defaults." Surprising for the user but safer if any v2 field would be actively dangerous to misinterpret.

The chosen behavior is asserted by `load_with_newer_version_succeeds` (Option A) — the test will need adjusting if Option B is picked.
